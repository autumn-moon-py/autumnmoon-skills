# SwiftUI 架构与状态归属

## 默认原则

- 新增原生页面优先使用 SwiftUI。
- 简单页面优先轻量结构，不为了“规范”强上厚重 ViewModel。
- 复杂页面需要共享状态、异步动作或多个服务协作时，再引入明确的状态对象。

## 属性包装器速查

| 包装器 | 适用场景 |
| --- | --- |
| `@State` | 视图自己拥有的本地状态 |
| `@Binding` | 子视图修改父视图状态 |
| `@Environment` | 读取环境值或共享对象 |
| `@Bindable` | 对注入的可观察对象建立双向绑定 |
| `let` | 只读注入的依赖或模型 |

规则：

- `@State` 默认使用 `private`。
- 传入的对象不要再声明成视图自有状态。
- 只读注入优先使用普通属性，不要滥用状态包装器。

## 结构建议

### 简单页面

适合：

- 只需本地状态。
- 少量异步请求。
- 逻辑容易解释。

做法：

- View 直接协调少量异步调用。
- 把重逻辑放进 Service / Client。

### 中复杂页面

适合：

- 多个异步动作。
- 页面状态明显复杂。
- 同时存在加载、错误、提交、刷新等流程。

做法：

- 定义显式状态对象。
- 明确页面拥有者与注入者。
- 让视图只消费状态与动作。

## 视图拆分规则

- 超过一个屏幕的复杂 `body`，优先拆成局部子视图。
- 复用意图明确时，提取独立 `View`。
- 只是局部结构且无复用时，可用计算属性或 `@ViewBuilder`。

## 环境注入

适合：

- 全局共享服务。
- 路由器。
- 会话、权限、主题等跨页面对象。

规则：

- 环境里放共享依赖，不放零散页面状态。
- 不要让环境对象变成“全项目万能仓库”。

## 版本门槛处理

- 使用较新 API 时，始终考虑 `#available` 与降级路径。
- 不要为了用新 API 破坏已有目标版本。

## 常见错误

- 把传入对象误写成 `@State`，导致外部更新失效。
- 页面只有几个状态，却引入过重的 ViewModel / Router 层。
- 所有页面都依赖一个巨大的环境对象。
- 一个 `body` 同时承载布局、数据加工、路由和业务判断。

## 快速检查

- [ ] 这个状态是谁拥有的。
- [ ] 这个依赖是注入的，还是视图自己创建的。
- [ ] 这个页面是否已经大到应该拆分。
- [ ] 是否需要 `#available` 与回退路径。
