# 性能排查与优化

## 处理顺序

1. 先看代码。
2. 再用 Instruments 取证。
3. 再做定点修改。
4. 最后复测。

不要一上来大改结构。

## 代码优先检查项

- 是否在 `body` 中做排序、过滤、格式化。
- 是否在 `body` 中同步解码图片或构造昂贵对象。
- `ForEach` 是否使用稳定身份。
- 是否存在顶层视图频繁切换导致身份抖动。
- 是否有宽范围状态变更引发整页更新。

## 常见热点

### 1. 重计算留在 `body`

处理：

- 预计算。
- 状态变化时更新缓存。
- 把重逻辑放到模型或服务层。

### 2. 列表身份不稳定

处理：

- 使用稳定 `id`。
- 不要对动态数据依赖 `\.self` 或瞬时索引。

### 3. 图片在主线程解码

处理：

- 提前下采样。
- 后台处理后再回主线程显示。

### 4. 广播式状态更新

处理：

- 缩小状态作用域。
- 拆分大对象。
- 让叶子节点只依赖自己真正需要的字段。

## Instruments 使用建议

- 尽量用 Release 构建。
- 能上真机就别只看模拟器。
- 复现最明确的慢路径：滚动、切页、动画、加载。
- 同时看 SwiftUI 模板和 Time Profiler。

## 优化优先级

1. 身份稳定性。
2. 主线程阻塞。
3. 状态更新范围。
4. 布局复杂度。
5. 动画与视觉细节。

## 常见误区

- 凭感觉说“应该是 SwiftUI 太慢”。
- 看见卡顿就先上 `Task` 或缓存，没有证据。
- 把性能问题和架构宗教混为一谈。

## 快速检查

- [ ] 有没有重计算留在 `body`。
- [ ] 列表身份是否稳定。
- [ ] 是否有主线程图片解码或昂贵格式化。
- [ ] 有没有基于 Instruments 的证据。
