# 网络层

## 默认方案

新代码默认使用 `URLSession` + `async/await`。

规则：

- 不新增无必要第三方网络框架。
- 请求、响应校验、解码、错误映射统一收口。

## 基本流程

1. 构造 `URLRequest`。
2. 发请求。
3. 校验 `HTTPURLResponse`。
4. 校验状态码。
5. 解码模型。
6. 映射错误。

## 状态码校验

`URLSession` 不会因为 4xx / 5xx 自动报错。

因此必须显式处理：

- 响应是不是 `HTTPURLResponse`
- 状态码是不是 `200..<300`

## 推荐结构

### Endpoint

负责：

- 路径。
- 方法。
- 查询参数。
- Header。

### API Client

负责：

- 统一构造请求。
- 执行请求。
- 响应校验。
- JSON 解码。
- 错误映射。

### Service / Feature Client

负责：

- 领域语义。
- 具体业务 API 调用。

## Token 刷新

如果存在 Token：

- 对 `401` 做一次受控刷新和一次重试。
- 不要在多个地方各自实现刷新逻辑。
- 不要让并发请求同时触发无序刷新。

## 设计要求

- Header、超时、解码策略统一配置。
- 错误类型是可区分、可展示、可记录的。
- 测试可以替换底层请求实现。

## 常见错误

- 状态码不校验就直接解码。
- 一部分请求走统一 Client，一部分页面里直接手写。
- `401` 刷新逻辑散落在多个 Service。
- 网络错误全都压成一个字符串。

## 快速检查

- [ ] 所有请求都经过统一 Client。
- [ ] 所有响应都做了状态码校验。
- [ ] 解码策略和错误映射一致。
- [ ] Token 刷新逻辑只有一个入口。
