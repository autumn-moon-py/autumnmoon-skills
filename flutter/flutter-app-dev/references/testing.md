# 测试策略

## 测试分层

### 单元测试

适合：

- Repository
- Service
- ViewModel / Controller
- 纯函数和转换逻辑

目标：

- 快。
- 稳定。
- 聚焦业务逻辑。

### Widget 测试

适合：

- 页面状态渲染。
- 关键交互。
- 控件显示与行为。

目标：

- 在不启动完整 App 的情况下验证 UI。

### 集成测试

适合：

- 路由流程。
- 登录、提交、下单、支付前确认等关键路径。
- Dart 与框架集成后的真实行为。

目标：

- 验证“多个部分一起工作”时是否正确。

## 测什么

### ViewModel / Controller

- 初始状态。
- 加载状态切换。
- 成功状态。
- 错误状态。
- 重试、刷新、分页等关键动作。

### Repository / Service

- 正常响应解析。
- 异常响应处理。
- 缓存命中与回退策略。

### Widget

- 加载、空态、错误态、成功态。
- 按钮点击是否触发预期动作。
- 列表、表单、弹窗等关键交互。

## Fake 优先

优先使用 `FakeRepository`、`FakeService` 等可控假实现，而不是到处堆 Mock。

优点：

- 输入输出更明确。
- 断言更直接。
- 测试更稳定。

## 命令建议

```bash
flutter test
flutter test test/features/profile/profile_view_model_test.dart
flutter test integration_test/app_test.dart
```

如果集成测试依赖模拟器或真机，先确保运行环境稳定，再跑流程。

## 最小覆盖建议

每个新功能至少包含：

1. 一个逻辑层测试。
2. 一个关键页面 Widget 测试。
3. 一条关键流程集成验证。

## 常见遗漏

- 只测成功态，不测错误态。
- 只测页面渲染，不测交互触发。
- 只测单页，不测跨页流程。
- 把异步状态变化写成脆弱的时间等待。

## 快速检查

- [ ] 逻辑层是否有单元测试。
- [ ] 页面关键状态是否有 Widget 测试。
- [ ] 核心流程是否有至少一条集成验证。
- [ ] 是否覆盖了失败路径、空态和重试路径。
