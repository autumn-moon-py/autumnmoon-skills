# 原生页面嵌入与 `PlatformView`

## 什么时候真的需要原生视图

优先判断：

- 这是不是系统级原生控件。
- Flutter 是否已经能稳定实现同样体验。
- 是否必须复用既有 iOS 视图或 SDK。

如果只是普通业务界面，不要因为“原生更顺手”就引入 `PlatformView`。

## 常见方式

### 方式 1：Flutter 调原生路由

适合：

- 原生页面本身就是独立流程。
- 不需要嵌入在 Flutter 布局树里。

优点：

- 生命周期更清楚。
- 性能和交互风险更低。

### 方式 2：`UiKitView` / `FlutterPlatformView`

适合：

- 必须把原生 `UIView` 嵌入 Flutter 页面。
- 地图、播放器、特殊输入控件等。

代价：

- 滚动、手势、渲染、裁切、层级更复杂。

### 方式 3：SwiftUI 先托管再暴露

适合：

- 原生界面本身使用 SwiftUI。
- 需要通过 UIKit 承载层接入 Flutter。

推荐思路：

- 先用 `UIHostingController` 承载 SwiftUI。
- 再将承载层暴露给 Flutter。

## 生命周期规则

- 视图创建、更新、销毁都要有明确入口。
- 原生监听、定时器、订阅在销毁时必须清理。
- 不要把 Flutter 页面生命周期和 iOS 控制器生命周期混为一谈。

## 线程与交互

- 原生 UI 更新必须在主线程。
- 手势冲突和滚动冲突要尽早验证。
- 复杂交互不要等联调末期才发现。

## 常见错误

- 普通业务界面也强行用 `PlatformView`。
- SwiftUI 页面直接裸暴露，没有中间承载层。
- 原生视图销毁时没释放监听和资源。
- 在滚动列表中大量嵌入原生视图，导致性能和手势异常。

## 快速检查

- [ ] 是否真的需要原生视图嵌入。
- [ ] 是否更适合走原生路由而不是 `PlatformView`。
- [ ] SwiftUI 页面是否先整理成稳定承载层。
- [ ] 生命周期与资源释放是否明确。
