# 联调、测试与排错

## 最小测试矩阵

### Dart 侧

验证：

- 调用入口是否正确。
- 参数序列化是否正确。
- 错误映射是否稳定。

### iOS 侧

验证：

- 通道是否注册成功。
- 原生服务是否收到调用。
- 主线程与后台线程是否符合要求。
- 原生 API 的成功与失败路径。

### 双端联调

验证：

- 从 Flutter 发起调用到 iOS 返回结果的整条链路。
- 关键错误场景是否能回到 Dart 并被正确展示。

## 排错顺序

1. 看注册是否成功。
2. 看 Flutter 侧是否发出了调用。
3. 看 iOS 侧是否收到调用。
4. 看参数是否正确反序列化。
5. 看线程是否正确。
6. 看返回值或错误是否正确回传。

## 日志建议

- Dart 侧打印方法名、参数摘要、错误码。
- iOS 侧打印注册时机、收到的调用、关键线程、返回结果。
- 日志要能对齐到一次完整请求，不要只打零碎信息。

## 高频故障清单

### 调用没有响应

优先检查：

- 通道名是否一致。
- 注册时机是否正确。
- Flutter Engine 是否为当前实例。

### 参数解析失败

优先检查：

- Dart 与 Swift 字段名是否一致。
- 空值语义是否一致。
- 枚举、时间、二进制数据是否有统一编码规则。

### iOS 页面不显示或交互异常

优先检查：

- 是否在主线程创建和更新 UI。
- `PlatformView` 是否存在尺寸或层级问题。
- 宿主控制器生命周期是否正确。

### 只在真机或只在模拟器失败

优先检查：

- 权限、签名、设备能力差异。
- 原生 SDK 是否依赖真实硬件。

## 回归验证建议

- 关键桥接能力至少保留一条自动化验证路径。
- Plugin 结构必须有 `example` 可快速验证。
- 联调通过后保留最小重现样例，避免后续只能人工复测。

## 快速检查

- [ ] Dart、iOS、双端联调都有验证入口。
- [ ] 出问题时有统一日志字段可追踪。
- [ ] 已经按注册、调用、参数、线程、返回值的顺序排查。
- [ ] `example` 或最小重现入口可长期复用。
