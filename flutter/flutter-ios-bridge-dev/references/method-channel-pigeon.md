# 通道选型：`MethodChannel`、`EventChannel`、`BasicMessageChannel`、`Pigeon`

## 选型原则

先看交互形态，再定通道。

| 场景 | 推荐 |
| --- | --- |
| 低频、请求-响应式调用 | `MethodChannel` |
| 持续事件流 | `EventChannel` |
| 自定义双向消息或非方法调用语义 | `BasicMessageChannel` |
| 长期维护、结构化、跨多人协作协议 | `Pigeon` |

## `MethodChannel`

适合：

- 调一次拿一次结果。
- 参数和返回值都不复杂。
- 功能边界清楚，生命周期短。

不适合：

- 高频数据推送。
- 协议会长期扩展。
- 需要强类型约束。

## `EventChannel`

适合：

- 传感器状态。
- 播放进度。
- 原生侧持续推送事件。

规则：

- 明确开始监听与停止监听。
- 原生侧释放资源必须和订阅生命周期一致。

## `BasicMessageChannel`

适合：

- 需要自定义消息语义。
- 双向交换非典型消息。

如果只是普通调用，不要为了“灵活”强上它。

## `Pigeon`

适合：

- 接口数量多。
- 参数模型复杂。
- 团队多人维护。
- 需要减少 Dart 与 Swift 手写协议偏差。

规则：

- 协议文件先于实现存在。
- 数据模型、错误语义和空值语义先定义清楚。

## 错误映射

桥接层必须统一错误格式：

- 错误码。
- 错误消息。
- 可选明细。

不要让 Dart 侧靠模糊字符串猜问题。

## 线程要求

- 涉及 iOS UI 的回调必须回主线程。
- 长耗时原生任务不要直接阻塞通道处理。

## 快速检查

- [ ] 我选的通道类型是否匹配交互形态。
- [ ] 协议是否有清晰的空值和错误语义。
- [ ] 高频事件是否误用了 `MethodChannel`。
- [ ] UI 相关回调是否遵守主线程要求。
