# 导航与界面骨架

## 路由选型

### `Navigator`

适合：

- 页面数量少。
- 无深链需求。
- 业务流程简单。

做法：

- 数据优先通过构造函数传递。
- 简单返回值通过 `Navigator.pop(result)` 回传。

### `go_router` 或统一路由层

适合：

- 有深链。
- 有 Web 同步需求。
- 导航结构复杂。
- 需要集中管理路由与守卫逻辑。

规则：

- 在同一项目里尽量保持一种主导航方式。
- 复杂应用不要一半 `pushNamed`，一半散落 `Navigator.push`。

### 嵌套路由 / 子导航

适合：

- 多步骤流程。
- 独立 Setup 向导。
- 每个 Tab 自己维护栈。

规则：

- 子流程用独立栈，不要把“步骤状态”硬塞进单页条件分支。
- 退出子流程时，统一处理返回和确认逻辑。

## 参数传递规则

- 优先通过构造函数传递明确类型数据。
- 避免把一大包动态参数挂在 `arguments` 上。
- 只在必须兼容既有路由系统时再使用弱类型参数包。

## Flutter 布局三原则

1. 约束向下传递。
2. 尺寸向上传递。
3. 父级决定子级位置。

写布局前先想清楚：

- 哪些区域固定尺寸。
- 哪些区域填充剩余空间。
- 哪些区域需要滚动。

## 响应式与自适应

### 响应式

目标：

- 在同一信息结构下，按可用空间调整布局比例。

典型手段：

- `LayoutBuilder`
- `Expanded`
- `Flexible`
- `Wrap`

### 自适应

目标：

- 按设备形态切换交互模式。

典型场景：

- 手机底部导航，平板侧边栏。
- 手机单列，平板双列。

## 常见布局问题与处理

### `ListView` 放进 `Column` 报无界高度

优先处理：

- 用 `Expanded` 包裹 `ListView`。
- 或者改成单一滚动容器。

### 内容溢出

优先检查：

- 是否缺少 `Expanded` / `Flexible`。
- 是否该滚动却没滚动。
- 是否写了过多固定宽高。

### 深层嵌套难维护

处理方式：

- 从外到内搭骨架。
- 提取独立局部 Widget。
- 保留一层一责，不把整个页面写成一个 500 行 `build`。

## 页面骨架建议

1. 先写 `Scaffold` 和一级布局。
2. 再写内容区域与滚动区域。
3. 再处理空态、错误态、加载态。
4. 最后补按钮、局部动画和细节样式。

## 快速检查

- [ ] 当前导航方案是否和复杂度匹配。
- [ ] 参数是否以强类型方式传递。
- [ ] 页面是否存在嵌套滚动冲突。
- [ ] 在窄屏和宽屏下是否都能稳定显示。
- [ ] 空态、错误态、加载态是否有独立布局。
