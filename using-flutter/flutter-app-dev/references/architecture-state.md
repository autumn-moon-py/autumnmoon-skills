# 架构与状态

## 目标

让 Flutter 页面、状态、数据源、服务边界清晰，避免把所有逻辑都塞进 Widget。

## 推荐结构

优先按功能分组，再在功能内部做轻量分层：

```text
lib/
  features/
    profile/
      data/
        profile_repository.dart
        profile_service.dart
      logic/
        profile_view_model.dart
      ui/
        profile_page.dart
        widgets/
          profile_header.dart
          profile_form.dart
```

如果现有项目已经固定为另一种结构，优先跟随现有约定，不在一个小任务里发起架构迁移。

## 状态选型

### 本地临时状态

适合用 `StatefulWidget` / `setState()` 的场景：

- 文本输入。
- 当前 Tab。
- 展开收起。
- 本地动画开关。
- 单个页面生命周期内有效的 UI 状态。

规则：

- 只在页面或组件内部使用。
- 不和业务数据源共用对象。
- 不为了“统一”而把局部状态抽成全局状态。

### 共享业务状态

适合放到 ViewModel / Controller / State Notifier 的场景：

- 多个组件共享的数据。
- 加载、错误、空态、成功态。
- 需要调用 Repository 或服务层的业务动作。
- 需要缓存、分页、刷新、提交等流程控制。

规则：

- 共享状态要有明确的单一事实来源。
- 状态对象只暴露 UI 需要的状态，不直接暴露底层细节。
- 新功能不要顺手引入新的状态管理流派。

## 默认职责划分

### Widget

负责：

- 展示状态。
- 收集用户输入。
- 触发 ViewModel / Controller 方法。
- 管理纯 UI 的临时状态。

不负责：

- 发请求。
- 写缓存。
- 业务规则判断。
- 组装多个数据源。

### ViewModel / Controller

负责：

- 接收交互动作。
- 组织页面状态。
- 调用 Repository。
- 对加载、错误、空态进行统一建模。

不负责：

- 直接访问底层平台 API。
- 直接耦合到 Widget 树细节。

### Repository

负责：

- 作为业务数据访问入口。
- 组合远端、本地、缓存和同步策略。
- 把原始响应转换为领域模型。

不负责：

- 控制页面级交互状态。
- 持有 Widget 相关上下文。

### Service / Data Source

负责：

- HTTP、数据库、文件、插件等具体访问。
- 原始请求与响应处理。

不负责：

- 页面语义。
- UI 状态。

## 实用规则

- ViewModel 里维护显式状态对象，而不是散落的十几个布尔值。
- Repository 返回明确模型，不把 `Map<String, dynamic>` 直接传给 UI。
- 对于一次性操作，优先定义 `loading / success / error` 语义，而不是在页面里拼接多个临时判断。
- 如果列表项只依赖自身数据，就把刷新范围压缩到列表项，避免整页重建。

## 常见反模式

### 1. Widget 里直接请求数据

问题：

- 生命周期难控。
- 重建可能触发重复请求。
- 测试困难。

### 2. 用一个大对象装下所有页面状态

问题：

- 改一个字段，整页甚至整模块都重建。
- 难以定位状态来源。

### 3. 业务规则写在按钮回调里

问题：

- 逻辑散落在多个页面。
- 无法复用和测试。

### 4. Repository 直接返回原始 JSON

问题：

- UI 需要知道后端字段细节。
- 一改接口就波及整个页面层。

## 快速检查

- [ ] 这个状态是局部状态，还是共享业务状态。
- [ ] 这个逻辑应该在 Widget、ViewModel 还是 Repository。
- [ ] 页面是否直接依赖了原始接口字段。
- [ ] 状态变化是否会引发不必要的整页重建。
