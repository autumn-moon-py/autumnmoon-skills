# Flutter 应用开发

## 概述

这个技能用于处理纯 Flutter 应用层的开发问题：需求拆解、页面组织、状态归属、导航流转、布局与主题、动画反馈，以及测试补齐。

如需具体模式，请按需读取以下引用资料：

- `references/architecture-state.md`
- `references/navigation-ui.md`
- `references/theme-animation.md`
- `references/testing.md`

## 适用场景

- 新增或重构纯 Flutter 页面。
- 梳理功能模块的分层和状态边界。
- 处理列表页、详情页、表单页、流程页的导航与布局。
- 统一主题、按钮、颜色、字体和动效。
- 为 Repository、ViewModel、Widget 或集成流程补测试。

## 核心规则

- 先划清边界，再写代码。页面、状态、数据源、导航职责不能混在一起。
- 优先遵循现有项目模式；没有既有约束时，采用轻量、可解释的方案，不为了小功能切换整套状态管理流派。
- 视图只表达状态，不在 Widget 树里堆业务逻辑和数据访问。
- 本地临时状态与共享状态分开处理。单个控件内部状态不要上提成全局状态。
- 先保证布局稳定，再追求花哨动效。溢出、抖动、不可滚动都属于基础问题。
- 主题和动效必须服务于交互，不为“看起来高级”引入无意义复杂度。
- 测试优先覆盖状态变化、关键流程和回归高风险点，而不是堆数量。

## 工作流

### 1. 判断任务边界

- 如果是纯 Flutter 业务页面，继续执行。

### 2. 先定分层与状态归属

- 明确哪些数据属于 Repository 或服务层。
- 明确哪些状态属于页面临时状态，哪些属于共享状态。
- 明确页面只消费状态，不直接负责数据源读写。

涉及分层、状态归属与文件拆分时，读取 `references/architecture-state.md`。

### 3. 再定导航与界面骨架

- 先确认页面之间的流转方式，再写按钮回调。
- 先搭稳定布局，再处理自适应、滚动与细节。
- 复杂流程使用独立的子导航或显式路由状态，不把整个流程塞进一个页面。

涉及路由、嵌套路由、响应式布局、滚动溢出时，读取 `references/navigation-ui.md`。

### 4. 统一视觉与交互反馈

- 全局视觉统一通过 `ThemeData` 和组件级主题完成。
- 动画先选最简单能满足需求的方案，只有需要显式控制时才上 `AnimationController`。
- 任何过重或不稳定的动效都先简化，再谈优化。

涉及主题、Material 3、按钮、转场、显式动画时，读取 `references/theme-animation.md`。

### 5. 补齐验证

- ViewModel、Repository、Service 做单元测试。
- 页面交互和渲染做 Widget 测试。
- 关键流程至少留一条集成测试。

涉及测试策略、Fake、Widget 测试、集成测试时，读取 `references/testing.md`。

## 常见错误

- 在 Widget 的 `build` 里直接请求网络或做重计算。
- 把临时 UI 状态提升成全局状态，导致整树重建。
- 页面跳转方案还没定，就先在多个按钮回调里硬编码 `Navigator`。
- 布局问题靠魔法数字硬顶，导致不同尺寸下溢出。
- 主题写一半全局、一半局部，最后颜色和字体无法统一。
- 明明只是简单状态切换，却直接上复杂状态管理框架。
- 只有金路径测试，没有异常路径、空状态和加载状态测试。

## 审查清单

- [ ] 这是纯 Flutter 任务，没有越界到原生桥接。
- [ ] 页面、状态、数据源职责边界清楚。
- [ ] Widget 树没有承载业务逻辑或数据访问。
- [ ] 布局在目标尺寸下无溢出、无不可滚动区域。
- [ ] 导航方式与业务复杂度匹配。
- [ ] 视觉统一通过主题管理，而不是零散写样式。
- [ ] 动画方案与需求复杂度匹配，没有过度设计。
- [ ] 单元、Widget、集成测试覆盖了主要回归风险。
