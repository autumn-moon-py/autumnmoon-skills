# Swift 并发

## 先确认上下文

在处理并发问题前，先确认：

- 当前是 Swift 6 还是更早版本。
- 是否启用了严格并发检查。
- 当前类型是否本来就属于 UI 主线程。

不要在不知道上下文时乱加注解。

## 最小安全修复原则

- UI 状态归 UI 线程：该上 `@MainActor` 的才上。
- 真正的后台工作才移出主线程。
- 能保持现有行为的前提下，优先选择最小改动。

## 常见问题与处理

### `@MainActor` 相关报错

先问自己：

- 这段代码是否真的属于 UI 状态。
- 调用方是否本来就应该在主线程。

如果答案是“不是”，不要用 `@MainActor` 掩盖问题。

### `Sendable` 报错

优先处理：

- 改成不可变值类型。
- 把跨线程共享的可变状态放进明确边界。
- 不轻易使用逃生口。

### 非结构化任务过多

优先处理：

- 能用结构化并发就不要随手起 `Task`。
- 有父子关系的任务，优先 `async let` 或 `TaskGroup`。

## 工具选择

| 需求 | 优先工具 |
| --- | --- |
| 单个异步流程 | `async/await` |
| 固定数量并行任务 | `async let` |
| 动态数量并行任务 | `withTaskGroup` |
| UI 更新 | `@MainActor` 或主线程切换 |

## 取消与生命周期

- 长任务必须考虑取消。
- 页面消失后，不要让旧任务继续回写页面状态。
- 重试、轮询、等待类逻辑需要明确退出条件。

## 常见错误

- 看到警告就无脑加 `@MainActor`。
- 用假 `await` 或无意义的异步包装绕过 lint。
- 任务早已不需要了，仍然继续回写状态。

## 快速检查

- [ ] 我知道当前代码的隔离边界。
- [ ] 我知道这段逻辑是否真的需要主线程。
- [ ] 我没有用注解掩盖数据竞争问题。
- [ ] 长任务和重试流程具备取消语义。
